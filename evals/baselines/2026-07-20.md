## Cross-scenario findings

The Builder Skills v1-absent outputs were strong at producing polished UI, focused automated tests, and server-only credential handling; the agents reported locally passing builds. The repeated weaknesses are architectural rather than basic implementation ability:

- design quality can arrive with an unnecessarily broad or decorative diff;
- provider familiarity does not reliably select the approved Project Secrets and manifest lifecycle;
- legacy capability vocabulary and paths reappear without explicit prohibition;
- fixture tests can support implementation confidence but cannot justify live-provider or domain-accuracy claims;
- a blocked API response is not the required user-facing safe preview and Project Secrets Setup flow.

Future integration skills must make official-document research mandatory while keeping `documentationUrl` optional. Generated code and `astropages/secrets.manifest.json` must land and be synchronized in the same exact generated-repository commit; synchronization ownership will be recorded by the current-flow audit.

These baselines are inputs to later skill design. They do not establish source/runtime parity, live provider correctness, route-manifest ownership, or a final granular-intent/internal-skill inventory.
